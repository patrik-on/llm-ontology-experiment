REPORT – Doterajšia práca na príprave datasetov a repozitára
Projekt: llm-ontology-experiment
Téma: LLM + Ontológie pre softvérové inžinierstvo, fine-tuning a neskôr RAG

================================================================================
1. CIEĽ DOTERAJŠEJ PRÁCE
================================================================================

Cieľom tejto fázy bolo pripraviť repozitár a dátovú pipeline pre prvú časť diplomovej práce zameranú na fine-tuning jazykových modelov pre dve úlohy softvérového inžinierstva:

1. Generovanie unit testov v Jave
2. Refaktoring Java kódu

Zároveň bolo cieľom pripraviť štruktúru tak, aby bola rozšíriteľná aj pre väčšie datasety a neskoršiu RAG vetvu.

V tejto fáze sme sa zatiaľ nesústredili na samotný tréning modelu, ale na:

- výber vhodných datasetov,
- stiahnutie a preskúmanie dát,
- návrh jednotného JSONL instruction formátu,
- prípravu spracovaných train/val/test datasetov,
- vytvorenie finálnych datasetov pre B1, B2-R a B2-T experimenty.

================================================================================
2. PÔVODNÁ ŠTRUKTÚRA REPOZITÁRA A JEJ ÚPRAVA
================================================================================

Na začiatku bola navrhnutá pomerne rozsiahla štruktúra repozitára, ktorá počítala s fine-tuningom aj budúcou RAG vetvou.

Po diskusii sme sa rozhodli začať kompaktnejšie a najprv pripraviť iba časť potrebnú pre fine-tuning. Štruktúra však ostala navrhnutá tak, aby sa dala neskôr rozšíriť o RAG, Graph RAG, embeddingy a ontológie.

Kľúčová dátová štruktúra po úpravách:

================================================================================
3. METHODS2TEST – TESTING DATASET
================================================================================

3.1 Charakter datasetu

Methods2Test je dataset určený na úlohu generovania unit testov. Obsahuje Java test cases a ich zodpovedajúce focal methods. Dataset je vhodný na supervised fine-tuning modelu, ktorý má z Java metódy generovať JUnit test.

Dataset obsahuje viacero reprezentácií vstupu:

- src_fm – iba focal method,
- src_fm_fc – focal method + focal class,
- src_fm_fc_co – focal method + focal class + constructors,
- src_fm_fc_ms – focal method + class + public method signatures,
- src_fm_fc_ms_ff – focal method + class + public methods + public fields.

Pre prvý prototyp sme zvolili najjednoduchšiu reprezentáciu:

input = src_fm
output = target

3.2 Stiahnutie datasetu

Dataset bol stiahnutý cez Git LFS priamo do:

data/raw/methods2test/

Použitý postup:

1. Vytvoril sa priečinok data/raw/.
2. Dataset sa naklonoval z GitHub repozitára microsoft/methods2test.
3. Spustil sa git lfs pull.
4. Overilo sa, že priečinky corpus/, dataset/ a methods2test-v1.0/ existujú.

3.3 Rozbalenie corpus/json

Použili sme časť:

data/raw/methods2test/corpus/json/

Nachádzali sa tam archívy:

- train.tar.bz2
- eval.tar.bz2
- test.tar.bz2

Tie boli rozbalené na:

- data/raw/methods2test/corpus/json/train/
- data/raw/methods2test/corpus/json/eval/
- data/raw/methods2test/corpus/json/test/

Každý split obsahoval priečinky podľa ID repozitárov a súbory typu:

994088_0_corpus.json

3.4 Štruktúra jedného záznamu

Ukážkový JSON obsahoval polia:

- target
- src_fm
- src_fm_fc
- src_fm_fc_co
- src_fm_fc_ms
- src_fm_fc_ms_ff

Pre prvú fázu bolo použité:

input = src_fm
output = target

3.5 Implementované súbory

Boli pripravené tieto súbory:

- scripts/inspect_methods2test.py
- scripts/prepare_methods2test.py
- src/llm_ontology/data/methods2test.py

3.6 Výstupný formát

Každý záznam bol prevedený do JSONL instruction formátu:

{
  "instruction": "Vygeneruj JUnit test pre nasledujúcu Java metódu.",
  "input": "<src_fm>",
  "output": "<target>",
  "domain": "testing",
  "source": "methods2test",
  "context_level": "src_fm",
  "source_file": "<pôvodný JSON súbor>"
}

================================================================================
4. MARV – KVALITNÝ REFACTORING DATASET
================================================================================

4.1 Charakter datasetu

MaRV je manuálne validovaný refactoring dataset. Obsahuje páry kódu pred a po refaktoringu z Java GitHub repozitárov. Dataset pokrýva štyri typy refaktoringu:

- Extract Method
- Rename Method
- Rename Variable
- Remove Parameter

Každý záznam obsahuje aj metadata a hodnotenia od reviewerov.

4.2 Stiahnutie a umiestnenie

MaRV bol stiahnutý ako replication package a rozbalený do:

data/raw/marv/

Hlavný datasetový súbor:

data/raw/marv/dataset/MaRV.json

4.3 Štruktúra MaRV.json

MaRV.json je slovník, kde hlavné kľúče sú typy refaktoringu:

- Extract Method
- Rename Method
- Rename Variable
- Remove Parameter

Každý záznam obsahuje najmä:

- refactoring_id
- commit_sha
- commit_link
- file_path
- description
- code_before
- code_after
- evaluations

Použité mapovanie:

input = code_before
output = code_after
refactoring_type = hlavný kľúč v MaRV.json

4.4 Implementované súbory

Boli vytvorené:

- src/llm_ontology/data/marv.py
- scripts/inspect_marv.py
- scripts/prepare_marv.py

4.5 Výstupný formát

Každý záznam bol prevedený na formát:

{
  "instruction": "Vygeneruj refaktorovanú verziu nasledujúceho Java kódu podľa typu refaktoringu: <refactoring_type>.",
  "input": "<code_before>",
  "output": "<code_after>",
  "domain": "refactoring",
  "source": "marv",
  "refactoring_type": "<refactoring_type>",
  "refactoring_id": "<id>",
  "commit_sha": "<commit_sha>",
  "commit_link": "<commit_link>",
  "file_path": "<file_path>",
  "description": "<description>",
  "evaluation_votes": [<votes>]
}

================================================================================
5. ML4REFACTORING – VEĽKÝ REFACTORING DATASET
================================================================================

5.1 Charakter datasetu

ML4Refactoring je väčší refactoring dataset obsahujúci source code pred a po refaktoringu. Použili sme ho ako väčší tréningový zdroj pre refactoring úlohu, aby bolo možné lepšie vyvážiť refactoring voči veľkému testing datasetu Methods2Test.

Dataset obsahuje:

- all.zip – source code before/after refactoring,
- refactoring50.sql.zip – štrukturálne metriky,
- projects-sample.xlsx – zoznam projektov.

5.2 Umiestnenie veľkých dát

Keďže all.zip má približne 40 GB, rozhodli sme sa ho neukladať do Git repozitára ani do OneDrive priečinka projektu.

Použitá lokálna cesta:

C:/datasets/ml4refactoring/all.zip
C:/datasets/ml4refactoring/all/

V repozitári ostali iba konfiguračné odkazy a spracované JSONL výstupy.

5.3 Štruktúra all.zip

Po rozbalení all.zip vznikla štruktúra:

C:/datasets/ml4refactoring/all/dataset/
├── apache-abdera.zip
├── apache-accumulo.zip
├── apache-activemq.zip
└── ...

Každý projektový ZIP má po rozbalení štruktúru:

output/
└── <project_name>/
    ├── error.txt
    ├── log.txt
    └── storage/
        └── <commit_sha>/
            ├── before-refactoring/
            └── after-refactoring/

5.4 Dôležité zistenie o súboroch

Súbory nemajú klasickú príponu .java, ale ich názov obsahuje .java, napríklad:

AbstractStreamWriter.java-2-Extract Method-589-startAuthor
AbstractStreamWriter.java-2-Extract Method-589-startAuthor-astc1
AbstractStreamWriter.java-2-Extract Method-589-startAuthor-astc2

Pre prvú verziu datasetu sme sa rozhodli použiť iba súbory, ktoré:

- obsahujú ".java" v názve,
- neobsahujú "-astc" v názve.

Súbory s -astc1 a -astc2 sa preskakujú.

5.5 Párovanie before/after

Pre každý commit sa párujú súbory podľa relatívnej cesty medzi:

before-refactoring/
after-refactoring/

Ak rovnaký relatívny súbor existuje v oboch priečinkoch, vytvorí sa jeden trénovací príklad.

5.6 Extrakcia typu refaktoringu

Typ refaktoringu sa extrahuje z názvu súboru ako substring. Podporované boli napríklad:

- Extract Method
- Move Method
- Pull Up Method
- Push Down Method
- Inline Method
- Rename Method
- Move Attribute
- Pull Up Attribute
- Push Down Attribute
- Rename Attribute
- Extract Class
- Move Class
- Rename Class
- Extract Interface
- Extract Superclass
- Remove Parameter
- Add Parameter
- Rename Variable

Ak sa typ nepodarí nájsť, použije sa hodnota:

unknown

5.7 Implementované súbory

Boli implementované:

- src/llm_ontology/data/ml4refactoring.py
- scripts/inspect_ml4refactoring.py
- scripts/prepare_ml4refactoring.py

Pipeline spracováva vždy jeden projektový ZIP naraz, rozbalí ho do dočasného priečinka, vyzbiera validné páry a potom dočasný priečinok odstráni. Tým sa predchádza zbytočnému rozbaľovaniu celého datasetu naraz.

5.8 Výstupný formát

Každý výstupný záznam má formát:

{
  "instruction": "Vygeneruj refaktorovanú verziu nasledujúceho Java kódu podľa typu refaktoringu: <refactoring_type>.",
  "input": "<obsah before-refactoring súboru>",
  "output": "<obsah after-refactoring súboru>",
  "domain": "refactoring",
  "source": "ml4refactoring",
  "project": "<project_name>",
  "commit_sha": "<commit_sha>",
  "file_path": "<relatívna cesta>",
  "refactoring_type": "<refactoring_type>"
}

================================================================================
9. DÔLEŽITÉ ROZHODNUTIA
================================================================================

1. Methods2Test sa používa cez corpus/json, nie cez dataset/ časť.

2. Pri Methods2Test sa používa context level src_fm, teda iba focal method.

3. Pri Methods2Test sa zachoval pôvodný train/eval/test split.

4. MaRV sa používa ako kvalitný manuálne validovaný refactoring dataset.

5. ML4Refactoring sa používa ako väčší refactoring tréningový zdroj.

6. ML4Refactoring all.zip je uložený mimo repozitára, aby sa predišlo problémom s veľkosťou, OneDrive synchronizáciou a Gitom.

7. ML4Refactoring sa nespracováva naraz celý, ale po jednotlivých projektových ZIPoch.

8. Súbory ML4Refactoring sa nehľadajú podľa prípony *.java, ale podľa toho, či názov obsahuje ".java".

9. Súbory obsahujúce "-astc" sa v prvej verzii preskakujú.

10. B1 combined dataset je vyvážený 1:1 medzi testing a refactoring doménou.

11. MaRV nie je v combined datasete, aby ostal ako samostatný benchmark.
