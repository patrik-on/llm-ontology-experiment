## Kontext

Dnešná práca nad projektom `llm-ontology-experiment` bola zameraná na prípravu a vyhodnotenie v2 modelov po oprave tréningovej pipeline. V2 modely boli natrénované po dôležitých zmenách v promptovaní a label maskingu:

- centralizovaný prompt formatter,
- jednotný prompt formát pre tréning aj evaluáciu,
- používanie datasetového `instruction` + `input`,
- EOS token na konci cieľového výstupu,
- label masking prompt tokenov na `-100`,
- bezpečné truncation správanie, aby output labels neboli celé odrezané,
- `DataCollatorForSeq2Seq` zachováva `labels=-100`.

Cieľom dňa bolo pripraviť a overiť v2 evaluation konfiguráciu, opraviť baseline inference problém, vytvoriť analýzu interferencie medzi shared a specialized modelmi a následne analyzovať samotné LoRA adaptéry.

## Oprava baseline modelu v inference

Pri spustení v2-only evaluácie sa objavil problém s baseline modelom. V `eval_models_v2_only.yaml` mal baseline nastavené:

```yaml
adapter_path: "baseline"
```

Inference kód to interpretoval ako reálnu cestu k PEFT adaptéru a pokúsil sa volať `PeftModel.from_pretrained(...)`. To spôsobilo chybu:

```text
ValueError: Can't find 'adapter_config.json' at 'baseline'
```

Oprava mala dve časti.

V konfigurácii sa baseline zmenil na rovnaký formát ako vo funkčnom `eval_models.yaml`:

```yaml
adapter_path: null
type: "baseline"
```

V kóde `src/llm_ontology/evaluation/inference_eval.py` pribudla pomocná funkcia:

```python
is_lora_model(model_config) -> bool
```

Tá zabezpečuje, že PEFT adapter sa načíta iba vtedy, keď ide o LoRA model alebo keď existuje reálny adapter path. Sentinel hodnoty ako `None`, `""`, `"baseline"`, `"none"` a `"null"` sa už nepovažujú za LoRA adaptér.

Tým sa baseline vyhodnocuje správne ako čistý base model cez:

```python
AutoModelForCausalLM.from_pretrained(...)
```

a bez volania:

```python
PeftModel.from_pretrained(...)
```

Bol pridaný aj regresný test pre túto logiku. Testovacia sada sa rozšírila z 17 na 18 testov.

### Metóda LoRA analýzy

Pre každý modul sa párovali LoRA váhy `lora_A` a `lora_B`.

Ak to rozmery umožnili, počítala sa aproximovaná LoRA update matica:

```text
DeltaW = B @ A
```

Zároveň sa počítala proxy norma:

```text
||A||_F * ||B||_F
```

Podobnosť adaptérov sa počítala ako cosine similarity medzi zodpovedajúcimi modulmi. Porovnávali sa iba spoločné LoRA moduly existujúce v oboch adaptéroch.

## Interpretácia dnešných výsledkov

Dnešné výsledky spolu vytvárajú pomerne jasný obraz:

1. Fine-tuning v2 výrazne pomáha oproti baseline.
2. Testing úloha profituje zo shared modelu takmer rovnako ako zo špecializovaného modelu.
3. Refactoring viac profituje zo špecializácie.
4. B2-T a B2-R LoRA adaptéry sú medzi sebou veľmi málo podobné.
5. B1 shared LoRA adaptér je celkovo podobnejší B2-T než B2-R.
6. B1 shared má top LoRA vrstvy, ktoré sa čiastočne prekrývajú s oboma špecializovanými adaptérmi.

Ak to preložíme do diplomovej interpretácie:

Testing a refactoring sa učia cez odlišné LoRA aktualizácie. Shared model dokáže zachytiť veľkú časť testing správania bez výraznej negatívnej interferencie. Pri refactoringu však shared model nedosiahne rovnakú kvalitu ako refactoring-specialized adaptér. To naznačuje, že refactoring úloha vyžaduje špecifickejšie adaptačné vzory alebo je citlivejšia na kapacitu LoRA adaptéra pri multi-domain fine-tuningu.

## Limity interpretácie

Tieto výsledky treba interpretovať opatrne:

- evaluation bola na limite 50, nie na celom test splite,
- testing aj refactoring metriky sú proxy metriky,
- LoRA cosine similarity váh nie je to isté ako funkčná podobnosť modelov,
- neanalyzovali sa aktivácie modelu,
- neanalyzovali sa attention heads priamo,
- zatiaľ chýba executable Java/JaCoCo subset,
- kvalitatívna manuálna analýza vybraných predikcií je stále potrebná.

Napriek tomu sú výsledky užitočné ako kombinácia výkonovej analýzy a reprezentational proxy analýzy.
